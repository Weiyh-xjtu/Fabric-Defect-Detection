# Day7 模型评估、调优与导出实施计划

## 目标
按照 `docs/` 中 Day7 文档，在保留现有未提交改动的前提下，补齐项目模型评估、调优与导出能力，并完成可执行验证。

## 成功标准
- Day7 文档中的评估流程与指标可通过项目接口/页面使用。
- Day7 文档中的调优参数真实传递到训练流程并被验证。
- Day7 文档要求的模型格式可以导出，产物可定位，失败信息明确。
- 新增或现有自动化测试通过；可运行路径完成端到端检查。
- 不覆盖用户在 `backend/app/training/training_service.py` 中的现有改动。

## 假设与约束
- 以 Day7 文档为需求真源；若文档与当前架构冲突，优先沿用现有架构并记录差异。
- 不进行与任务无关的重构。
- 不提交、不推送代码，除非用户另行要求。
- 规划文件仅记录可信的项目决策；文档摘录与代码发现写入 `findings.md`。

## 决策记录
| 决策 | 原因 |
|---|---|
| 先做需求—实现差距分析，再编码 | 避免猜测 Day7 的接口与交互要求 |
| 复用 `TrainingTask`/`ModelVersion`，不新增表 | 现有字段已覆盖 Day7 持久化需求 |
| 提取统一训练产物路径与评估结果解析函数 | 服务、下载、预测和独立脚本需避免重复且保持一致 |
| 调优采用现有训练表单扩展增强参数 | Day7 未要求自动参数搜索，最小实现是让已有调优参数真实生效 |
| 新接口执行任务所有权校验 | 防止鉴权用户访问其他用户模型产物 |

## 遇到的错误
| 错误 | 尝试次数 | 解决方案 |
|---|---:|---|
| `Read` 传入空 `pages` 参数导致校验失败 | 1 | 改为有效页码参数后成功读取 |
| 使用含花括号的 `Glob` 盘点源码未匹配到文件且扫入 `node_modules` | 1 | 改用具体目录和简单扩展名模式 |
| 更新 `findings.md` 时使用了已被替换的旧文本 | 1 | 重新读取文件后按当前内容定位插入点 |
| 系统 Python 3.13 未安装 `pytest` | 1 | 查找并使用项目虚拟环境；若不存在则改做可用的静态/前端验证并明确说明 |
| 系统 Python 仅有 3.13，且未安装 FastAPI/SQLAlchemy 等后端依赖 | 1 | 无法在当前环境启动后端；保留 compileall 验证并明确限制 |
