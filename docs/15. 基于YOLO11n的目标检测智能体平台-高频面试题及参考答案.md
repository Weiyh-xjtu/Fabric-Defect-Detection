## 第一部分：项目答辩高频提问

### 1.1 项目概述类

#### Q1：请简要介绍一下你的项目

本项目是「基于 YOLO11n 与多智能体架构的遥感目标检测平台」（RSOD Agent Platform），是一个将计算机视觉（CV）与大语言模型（LLM）深度融合的全栈 AI 应用。

**核心能力包括：**

- 目标检测：支持单图、批量、ZIP 压缩包、视频、摄像头实时检测五种模式，基于 YOLO11n 模型
- 智能对话：基于 LangChain/LangGraph 构建的多智能体系统，支持单 Agent 和多 Agent 两种模式
- RAG 知识库：检索增强生成，支持遥感领域专业知识问答
- 模型训练：完整的 YOLO11n 模型训练、评估、导出全流程管理
- 数据看板：检测数据的统计分析与可视化

**技术栈**：后端 Python + FastAPI，前端 Vue 3 + Element Plus，数据库 PostgreSQL + pgvector，缓存 Redis，对象存储 MinIO，AI 框架 LangChain + LangGraph。

#### Q2：你为什么选择这个课题？项目的创新点在哪里？

**选题原因：**

- 遥感目标检测在国防、城市规划、环境监测等领域有重要应用价值
- 传统检测系统操作门槛高，需要将 AI 能力与自然语言交互结合，降低使用门槛

**创新点：**

1. CV + LLM 深度融合：不是简单叠加，而是通过 Agent 工具调用机制，让大模型自主决定何时调用检测工具，实现"对话即检测"
2. 多智能体协作架构：基于 LangGraph 状态图，将检测、分析、问答拆分为独立 Agent，由 Supervisor 统一调度，职责清晰、可扩展
3. RAG 知识库增强：将遥感领域专业知识向量化存储到 pgvector，实现基于检索的知识问答，避免大模型幻觉
4. 五种检测模式一体化：单图/批量/ZIP/视频/摄像头，覆盖实际应用的各类场景

#### Q3：项目的技术架构是怎样的？

```
前端 (Vue 3) ──HTTP/WebSocket──▶ FastAPI 后端 ──▶ PostgreSQL + pgvector
                                      │
                                      ├──▶ Redis（缓存/记忆/限流）
                                      ├──▶ MinIO（对象存储）
                                      ├──▶ YOLO 模型（推理/训练）
                                      └──▶ LLM API（通义千问/DeepSeek）
```

**后端采用分层架构：**

- API 层：FastAPI 路由，处理 HTTP/WebSocket 请求
- Agent 层：LangChain/LangGraph 智能体，负责意图识别和工具编排
- Service 层：业务逻辑（检测、训练、看板等）
- Entity 层：SQLAlchemy ORM 数据模型（15 张表）
- Storage 层：Redis、MinIO、pgvector 等外部存储封装

### 1.2 核心技术实现类

#### Q4：多智能体架构是如何实现的？

多 Agent 模式基于 LangGraph 状态图实现，核心架构为：

```
用户输入 → Supervisor（意图路由）→ Detection/Analysis/QA Agent → Summarize（汇总）→ 回复
```

**具体实现：**

1. 共享状态（AgentState）：使用 TypedDict 定义，包含 messages、next_agent、各 Agent 结果等字段，通过 `add_messages` reducer 自动追加消息
2. Supervisor Agent：接收用户输入，通过 LLM 进行意图识别，输出路由决策（detection/analysis/qa），使用 `add_conditional_edges` 实现条件路由
3. 三个子 Agent：
   - Detection Agent：调用检测工具执行目标检测
   - Analysis Agent：查询检测历史和统计数据
   - QA Agent：检索知识库回答专业问题
4. 汇总节点：整合各 Agent 结果，结合对话历史，通过 LLM 生成最终回复

代码实现在 `graph.py` 中通过 `StateGraph(AgentState)` 构建状态图，`workflow.compile()` 编译为可执行图。

#### Q5：单 Agent 和多 Agent 模式有什么区别？各适用什么场景？

| 维度 | 单 Agent 模式 | 多 Agent 模式 |
| --- | --- | --- |
| 框架 | LangChain AgentExecutor | LangGraph StateGraph |
| 决策方式 | LLM 自主选择工具链 | Supervisor 路由 + 子 Agent 分工 |
| 工具数量 | 绑定全部 7 个工具 | 各 Agent 只绑定自己领域的工具 |
| 流式输出 | SSE 流式（thinking/text_chunk/tool_start/tool_end） | SSE 流式 |
| 适用场景 | 简单任务、工具少时 | 复杂任务、需要职责分离时 |
| 可扩展性 | 工具多了 LLM 选择准确率下降 | 新增 Agent 不影响现有逻辑 |

单 Agent 适合快速原型和工具数量较少时；多 Agent 适合复杂业务场景，每个 Agent 职责单一、提示词更聚焦，LLM 决策准确率更高。

#### Q6：RAG 知识库是怎么实现的？

RAG 全流程分为索引构建和检索问答两个阶段：

**索引构建（离线）：**

1. 文档加载：读取 `knowledge_base/` 目录下的 Markdown 文件（遥感知识、YOLO 基础、评估指标）
2. 文本分块：自定义分块器，500 字符/块，50 字符重叠，保留标题层级上下文
3. 向量化：调用通义千问 Embedding API（1024 维）
4. 向量存储：存入 PostgreSQL pgvector 扩展，使用 IVFFlat 索引

**检索问答（在线）：**

1. 用户提问 → Embedding API 向量化查询
2. pgvector 余弦相似度检索，Top-K=3，相似度阈值 0.5
3. 检索结果格式化为上下文
4. 上下文 + 用户问题 → LLM 生成最终回答

**关键设计：**

- 50 字符重叠保证语义连续性
- 保留标题层级作为 metadata，便于溯源
- 相似度阈值过滤低质量结果
- 支持强制重建索引（`force_rebuild=True`）

#### Q7：SSE 流式对话是怎么实现的？

SSE（Server-Sent Events）是一种基于 HTTP 的单向推送协议。本项目的实现方式：

**后端（FastAPI）：**

```python
async def chat_stream():
    async def event_generator():
        # 1. thinking 事件：LLM 正在思考
        yield f"data: {json.dumps({'type': 'thinking'})}\n\n"
        # 2. 流式调用 LLM
        async for event in agent.astream_events(...):
            if event_type == "text_chunk":
                yield f"data: {json.dumps({'type': 'text_chunk', 'content': chunk})}\n\n"
            elif event_type == "tool_start":
                yield f"data: {json.dumps({'type': 'tool_start', 'tool': name})}\n\n"
        # 3. done 事件
        yield f"data: {json.dumps({'type': 'done'})}\n\n"
    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

**前端**：使用 EventSource 或 fetch + ReadableStream 接收事件流，根据事件类型更新 UI（显示思考动画、逐字输出文本、显示工具调用状态）。

**事件类型**：thinking、text_chunk、tool_start、tool_end、done、error

#### Q8：视频检测是如何实现的？

视频检测的核心思路是帧采样 + 场景变化检测 + 逐帧推理：

1. 视频解码：OpenCV 打开视频，获取总帧数、fps、分辨率
2. 帧采样策略：根据 max_frames（默认 50）和视频总帧数动态计算采样间隔 `effective_interval = max(frame_sample_rate, total_frames // max_frames)`，确保帧均匀分布
3. 场景变化检测：将每帧缩小到 100×100 灰度图，与前帧做 absdiff，差值均值 > 10 则判定场景变化
4. 智能推理：仅在场景变化时执行 YOLO 推理，场景未变化时复用上一帧的检测结果，节省计算资源
5. 标注视频合成：每帧绘制检测框后写入视频，最后通过 ffmpeg 转码为 H.264（浏览器兼容）
6. 结果上传：标注视频上传到 MinIO，检测结果持久化到 PostgreSQL

**优化点：**

- 场景变化检测避免对静止画面重复推理
- 动态采样间隔适应不同时长的视频
- ffmpeg H.264 转码确保浏览器可播放

#### Q9：摄像头实时检测是如何实现的？

摄像头实时检测基于 WebSocket 双向通信：

1. 前端：调用浏览器 getUserMedia API 获取摄像头视频流，定时抓帧（如每秒 2-3 帧），将图像编码为 Base64 通过 WebSocket 发送
2. 后端：接收 Base64 图像 → 解码 → YOLO 推理 → 生成标注图 → 编码为 Base64 → 通过 WebSocket 返回
3. 前端：接收标注图并实时显示，同时统计 FPS

**关键设计：**

- WebSocket 相比 HTTP 避免了反复建立连接的开销
- 前端控制抓帧频率，避免后端过载
- 后端模型使用单例缓存，避免重复加载
- 支持实时调整置信度和 IoU 阈值

#### Q10：模型训练功能是如何实现的？

模型训练基于 Ultralytics YOLO11 框架，实现了完整的训练管理流程：

1. 任务创建：用户选择模型规格（n/s/m/l/x）、配置超参数（epochs、batch_size、optimizer、learning_rate）
2. 数据准备：支持 COCO/VOC 格式转换为 YOLO 格式，自动划分训练集/验证集/测试集（8:1:1）
3. 异步训练：后台线程执行训练，不阻塞 API 响应
4. 指标监控：每个 epoch 的 box_loss、cls_loss、dfl_loss、mAP、precision、recall 实时持久化到 `training_metrics` 表
5. 模型评估：支持验证集和测试集评估，输出 per-class AP
6. 模型导出：训练完成的模型注册到 `model_versions` 表，支持标记为默认模型，权重文件存储到 MinIO
7. 模型下载：支持下载训练好的模型权重

### 1.4 安全与性能类

#### Q12：项目中的安全机制有哪些？

1. 认证：JWT Bearer Token，OAuth2 密码模式，Token 默认 30 分钟过期
2. 密码安全：bcrypt 加密存储，72 字节截断防止 DoS 攻击
3. RBAC 权限控制：用户-角色-权限三级模型，admin/operator/viewer 三种角色
4. API 速率限制：基于 Redis 的分布式限流中间件，不同接口不同限流规则（如单图检测 30 次/分钟，视频检测 5 次/分钟）
5. CORS 白名单：可配置的跨域请求白名单
6. 全局异常处理：系统异常不暴露内部堆栈信息
7. 操作审计：`operation_logs` 表记录关键操作

#### Q13：限流中间件是怎么实现的？

基于 Redis 的固定窗口速率限制算法：

1. 中间件拦截每个请求，获取客户端 IP 和请求路径
2. 根据路径匹配限流规则（如 `/api/detection/single` 限制 30 次/分钟）
3. 在 Redis 中以 `rate_limit:{path}:{ip}` 为 key，设置过期时间（窗口期）
4. 每次请求计数 +1，超过限制返回 429 Too Many Requests
5. 支持 X-Forwarded-For 和 X-Real-IP 获取真实 IP（兼容反向代理）

**不同接口的限流策略：**

- 单图检测：30 次/分钟
- 批量检测：10 次/分钟
- 视频检测：5 次/分钟
- 摄像头检测：10 次/分钟
- 其他接口：60 次/分钟

### 1.5 项目难点与优化类

#### Q14：项目中遇到了哪些技术难点？如何解决的？

**难点 1：Agent 工具调用准确率低**

- 问题：单 Agent 绑定 7 个工具时，LLM 经常选错工具
- 解决：将提示词集中管理（`prompts.py`），明确工具调用优先级规则；多 Agent 模式下每个 Agent 只绑定领域相关工具，提高准确率

**难点 2：视频检测性能瓶颈**

- 问题：逐帧推理耗时过长，一个 30 秒视频可能需要数分钟
- 解决：引入场景变化检测，只在画面变化时执行推理；动态帧采样避免密集采样

**难点 3：RAG 检索质量不稳定**

- 问题：文本分块边界切断语义，导致检索结果不完整
- 解决：50 字符重叠保证语义连续；保留标题层级作为 metadata；设置相似度阈值 0.5 过滤低质量结果

**难点 4：模型加载慢导致首次检测延迟高**

- 问题：YOLO 模型每次推理都重新加载
- 解决：实现模型单例缓存（`_model_cache` 字典），同一模型只加载一次

**难点 5：流式对话与工具调用的协调**

- 问题：SSE 流式输出中，工具调用是异步的，需要正确编排事件顺序
- 解决：使用 LangChain 的 `astream_events` API，区分 thinking/text_chunk/tool_start/tool_end 等事件类型

#### Q15：项目还有哪些可以优化的地方？

1. Agent 方面：引入 ReAct（Reasoning + Acting）模式，让 Agent 在调用工具前先输出推理过程，提高可解释性
2. RAG 方面：引入 Hybrid Search（关键词检索 + 语义检索混合），提高检索召回率；支持 Reranker 对检索结果重排序
3. 检测性能：引入 TensorRT 或 ONNX Runtime 加速推理；支持 GPU 推理和模型量化（INT8/FP16）
4. 视频检测：引入目标跟踪算法（如 ByteTrack/DeepSORT），实现跨帧目标 ID 一致性和精确计数
5. 前端：引入 WebSocket 心跳机制防止连接断开；Web Worker 处理图像编码避免主线程阻塞
6. 限流：从固定窗口升级为滑动窗口或令牌桶算法，更平滑地限流
7. 可观测性：接入 Prometheus + Grafana 监控系统指标，接入 Jaeger 做分布式链路追踪

## 第二部分：考研复试面试高频题

### 2.1 计算机视觉基础

#### Q16：什么是目标检测？与图像分类有什么区别？

**图像分类**：判断一张图片属于哪个类别（整图一个标签），如"这是一张猫的图片"。

**目标检测**：不仅要识别图片中有什么目标，还要定位目标的位置（用边界框 Bounding Box 表示），即同时完成分类 + 定位。输出格式通常为 `(class, confidence, x1, y1, x2, y2)`。

**目标检测的难点：**

- 目标尺度变化（大目标和小目标共存）
- 目标遮挡和重叠
- 背景复杂
- 实时性要求

本项目中使用 YOLO11n 进行遥感图像目标检测，检测四类目标：飞机、油罐、立交桥、操场。

#### Q17：什么是 YOLO？YOLO 的核心思想是什么？

YOLO（You Only Look Once）是一种单阶段（One-Stage）目标检测算法，核心思想是将目标检测转化为回归问题。

**核心思想：**

1. 将输入图像划分为 S×S 的网格
2. 每个网格直接预测边界框（bbox）和类别概率
3. 一次前向传播即可得到所有目标的位置和类别

**与 Two-Stage 方法（如 Faster R-CNN）的对比：**

| 维度 | YOLO (One-Stage) | Faster R-CNN (Two-Stage) |
| --- | --- | --- |
| 流程 | 直接回归 bbox + 类别 | 先提候选框，再分类回归 |
| 速度 | 快，适合实时检测 | 慢，精度高 |
| 小目标 | 相对较弱 | 相对较好 |

**YOLO11n 的改进：**

- 更高效的骨干网络（Backbone）
- 改进的特征金字塔（Neck）
- 解耦头（Decoupled Head）分别预测分类和回归
- Anchor-Free 设计，减少超参数

#### Q18：什么是 IoU？什么是 NMS？

**IoU（Intersection over Union，交并比）：**

IoU = 预测框与真实框的交集面积 / 预测框与真实框的并集面积

- 取值范围 [0, 1]，值越大表示定位越准确
- 通常 IoU ≥ 0.5 认为检测正确
- 用于评估检测精度，也用于 NMS 中去除重叠框

**NMS（Non-Maximum Suppression，非极大值抑制）：**

作用：去除冗余的重叠检测框，只保留最好的那个。

算法流程：

1. 按置信度降序排列所有检测框
2. 选置信度最高的框，保留
3. 计算其余框与该框的 IoU，去除 IoU > 阈值的框
4. 重复步骤 2-3，直到所有框处理完毕

本项目中 NMS IoU 阈值默认设置为 0.45。

#### Q19：什么是 mAP？如何计算？

mAP（mean Average Precision，平均精度均值）是目标检测最常用的评估指标。

**计算流程：**

1. Precision（精确率）= TP / (TP + FP)，检测正确的目标数 / 总检测数
2. Recall（召回率）= TP / (TP + FN)，检测正确的目标数 / 实际总目标数
3. P-R 曲线：以 Recall 为横轴，Precision 为纵轴绘制的曲线
4. AP（Average Precision）：P-R 曲线下的面积（AUC），衡量单个类别的检测精度
5. mAP：所有类别 AP 的平均值

**常用变体：**

- mAP@0.5：IoU 阈值为 0.5 时的 mAP（VOC 标准）
- mAP@0.5:0.95：IoU 从 0.5 到 0.95（步长 0.05）的 mAP 平均值（COCO 标准，更严格）

本项目中训练时会同时监控 box_loss、cls_loss、dfl_loss、mAP、precision、recall 等指标。

#### Q20：什么是混淆矩阵？精确率和召回率有什么关系？

**混淆矩阵（Confusion Matrix）：**

|  | 预测为正 | 预测为负 |
| --- | --- | --- |
| 实际为正 | TP（真正例） | FN（假反例） |
| 实际为负 | FP（假正例） | TN（真反例） |

**精确率（Precision）= TP / (TP + FP)**：预测为正的里面有多少是真的

**召回率（Recall）= TP / (TP + FN)**：真的正例中有多少被找到了

**关系**：Precision 和 Recall 通常是矛盾的：

- 提高置信度阈值 → Precision 升高，Recall 下降（漏检增多）
- 降低置信度阈值 → Recall 升高，Precision 下降（误检增多）

F1-Score = 2 × Precision × Recall / (Precision + Recall)，是两者的调和平均值。

本项目中置信度阈值默认 0.25，可根据实际需求调整以平衡精确率和召回率。

### 2.2 深度学习与模型训练

#### Q21：什么是过拟合？如何防止过拟合？

**过拟合**：模型在训练集上表现很好，但在验证集/测试集上表现差，说明模型"记住"了训练数据而不是学到了通用规律。

**防止过拟合的常用方法：**

1. 数据增强：翻转、旋转、缩放、裁剪、颜色抖动等（YOLO 内置 Mosaic、MixUp）
2. 正则化：L1/L2 正则化、Dropout
3. 早停（Early Stopping）：监控验证集 loss，不再下降时停止训练
4. 增加数据量：更多训练数据
5. 减小模型复杂度：使用更小的模型（如 yolo11n 而不是 yolo11x）
6. 学习率调度：余弦退火、WarmUp 等策略

在本项目中，通过观察训练指标（train_loss 下降但 val_loss 上升）可以判断是否过拟合，并支持 early stopping 和超参数调优。

#### Q22：YOLO 训练中的 loss 有哪些？分别代表什么？

YOLO11n 训练中涉及三种 loss：

1. box_loss（边界框回归损失）：衡量预测框与真实框的位置偏差，使用 CIoU Loss 或 DFL（Distribution Focal Loss）
2. cls_loss（分类损失）：衡量目标类别预测是否正确，使用 BCE（Binary Cross-Entropy）Loss
3. dfl_loss（分布焦点损失）：DFL 将边界框回归建模为概率分布，提高对模糊边界的定位精度

**训练时关注：**

- 三种 loss 都应该随 epoch 递减并趋于稳定
- 如果 train loss 持续下降但 val loss 上升，说明过拟合
- mAP 是最综合的评估指标，应在验证集上持续上升

#### Q23：什么是迁移学习？在目标检测中如何应用？

**迁移学习**：将在一个任务上学到的知识迁移到另一个相关任务上，减少对大量标注数据的依赖。

**在目标检测中的应用：**

1. 预训练权重：使用在 COCO 数据集上预训练的 YOLO 权重作为初始化
2. Fine-tuning：冻结骨干网络（Backbone），只训练检测头（Head）；或者全部层以小学习率微调
3. 场景适配：遥感图像与日常图像差异大，需要用自己的数据集微调

**本项目的模型回退策略体现了迁移学习思想：**

- 优先使用针对特定场景训练的模型（`model_versions` 中 `is_default=True`）
- 找不到则使用最新训练产出的模型
- 最终回退到预训练模型 `yolo11n.pt`

### 2.3 大语言模型与 Agent

#### Q24：什么是 LangChain？什么是 LangGraph？

**LangChain** 是一个用于构建 LLM 应用的框架，核心抽象包括：

- Model I/O：统一的 LLM 调用接口（支持 OpenAI、通义千问、Ollama 等）
- Tools：让 LLM 调用外部函数（如搜索、计算、数据库查询）
- Agent：LLM 自主决定调用哪些工具、按什么顺序执行
- Memory：对话记忆，支持跨轮次上下文

**LangGraph** 是 LangChain 团队推出的多 Agent 编排框架，基于状态图（StateGraph）：

- Node（节点）：执行函数，如 Agent 调用、工具执行
- Edge（边）：节点间的流转关系
- Conditional Edge（条件边）：根据状态动态决定下一个节点
- State（状态）：TypedDict 定义的共享状态容器，在节点间流转

本项目用 LangGraph 构建 Supervisor → Detection/Analysis/QA → Summarize 的多 Agent 状态图。

#### Q25：什么是 RAG？为什么需要 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与大模型生成相结合的技术。

**为什么需要 RAG：**

1. 减少幻觉：大模型可能编造不存在的信息，RAG 基于真实文档回答
2. 知识时效性：大模型训练数据有截止日期，RAG 可以检索最新文档
3. 领域专业性：通用大模型缺乏垂直领域知识，RAG 可以注入专业知识库
4. 可溯源：答案可以追溯到具体文档片段，增强可信度

**RAG 基本流程：**

```
索引：文档 → 分块 → 向量化 → 存入向量数据库
问答：查询 → 向量化 → 相似度检索 → 取回相关文档片段 → 拼接为 Prompt → LLM 生成回答
```

本项目中，将遥感领域知识（YOLO 基础、评估指标、遥感概述）向量化存入 pgvector，实现专业知识问答。

#### Q26：什么是向量数据库？pgvector 和专用向量数据库（如 Milvus）有什么区别？

**向量数据库**：专门用于存储和检索高维向量的数据库，支持相似度搜索（余弦相似度、欧氏距离等）。

**pgvector vs 专用向量数据库：**

| 维度 | pgvector | Milvus/Pinecone |
| --- | --- | --- |
| 部署复杂度 | 低，PostgreSQL 扩展 | 高，独立服务 |
| 适用规模 | 百万级以下 | 十亿级以上 |
| 查询能力 | SQL + 向量混合查询 | 纯向量查询为主 |
| 索引类型 | IVFFlat、HNSW | 多种（IVF、HNSW、DiskANN 等） |
| 事务支持 | 完整 ACID | 有限 |
| 运维成本 | 低（复用 PostgreSQL） | 高（独立集群） |

**本项目选择 pgvector 的原因：**

- 知识库规模小（几十到几百条向量），pgvector 完全够用
- 复用已有的 PostgreSQL 实例，无需额外部署
- 支持 SQL 混合查询，便于管理 metadata

#### Q27：什么是 Embedding？文本向量化的过程是怎样的？

**Embedding（嵌入/向量化）**：将文本转换为固定维度的稠密向量，使语义相近的文本在向量空间中距离相近。

**过程：**

1. 输入文本 → Tokenizer 分词 → Token ID 序列
2. Token ID 序列 → Embedding 层 → Token 向量序列
3. Token 向量序列 → Transformer 编码 → 上下文感知的向量序列
4. 向量序列 → Pooling（如 `[CLS]` token 或 mean pooling）→ 固定维度的文本向量

**本项目使用的 Embedding 模型：**

- 通义千问 text-embedding-v2（1024 维），通过阿里云 DashScope API 调用
- 支持批量向量化（`embed_texts`）和单条查询向量化（`embed_query`）

**关键参数：**

- 维度：1024（越高表达力越强，但存储和计算成本也越高）
- 相似度度量：余弦相似度（衡量方向相似性，不受向量长度影响）

#### Q28：Agent 的 Tool Calling 机制是如何工作的？

Tool Calling 是让 LLM 能够调用外部函数的机制，工作流程：

1. 工具定义：用 JSON Schema 描述工具的名称、参数、功能

   ```python
   @tool
   def detect_single_image(image_path: str, user_id: int) -> dict:
       """对单张遥感图片执行目标检测"""
       ...
   ```

2. 工具绑定：将工具列表绑定到 LLM

   ```python
   agent = create_openai_tools_agent(llm, tools, prompt)
   ```

3. LLM 决策：用户输入后，LLM 分析意图，决定是否需要调用工具，以及调用哪个工具、传什么参数
4. 工具执行：框架自动执行 LLM 选择的工具，获取返回结果
5. 结果回传：工具结果回传给 LLM，LLM 生成最终回复

**核心优势**：LLM 不直接执行操作，而是作为"大脑"做决策，工具作为"手脚"执行具体任务，实现了推理与执行的解耦。

### 2.4 软件工程与系统设计

#### Q29：什么是 RESTful API？你的项目是如何设计 API 的？

RESTful API 是一种基于 HTTP 协议的 API 设计风格，核心原则：

- 资源导向：URL 表示资源（名词），HTTP 方法表示操作（动词）
- 无状态：每个请求包含所有必要信息，服务器不保存会话状态
- 统一接口：GET（查询）、POST（创建）、PUT（更新）、DELETE（删除）

**本项目的 API 设计：**

- 所有接口以 `/api` 为前缀
- 使用 FastAPI 框架，自动生成 Swagger 文档（`/docs`）和 ReDoc（`/redoc`）
- 请求/响应使用 Pydantic Schema 做数据验证和序列化
- 统一错误响应格式
- SSE 用于流式对话，WebSocket 用于实时摄像头检测

**示例：**

- `POST /api/detection/single` — 单图检测
- `GET /api/dashboard/stats` — 获取看板统计
- `POST /api/auth/login` — 用户登录

#### Q30：什么是 JWT？JWT 的工作原理是什么？

JWT（JSON Web Token）是一种紧凑的、自包含的 Token 标准（RFC 7519），用于在各方之间安全传输信息。

**结构（三段式，用 `.` 分隔）：**

```
Header.Payload.Signature
```

- Header：算法类型（如 HS256）
- Payload：载荷数据（如 `{"sub": user_id, "exp": timestamp}`）
- Signature：签名 = HMAC-SHA256(Header + Payload, Secret)

**工作原理：**

1. 用户登录 → 服务器验证密码 → 生成 JWT（包含 user_id 和过期时间）→ 返回给前端
2. 前端将 JWT 存储在 localStorage/cookie 中
3. 后续请求在 Header 中携带 `Authorization: Bearer <token>`
4. 服务器验证签名和过期时间 → 从 Payload 中提取 user_id → 处理请求

**JWT 的优缺点：**

- 优点：无状态（服务器不需要存储 session）、易于分布式部署
- 缺点：Token 签发后无法主动撤销（除非引入黑名单机制）、Payload 是 Base64 编码不是加密

本项目中 JWT 默认 30 分钟过期，使用 HS256 算法签名。

#### Q31：什么是 RBAC 权限模型？

RBAC（Role-Based Access Control，基于角色的访问控制）是一种通过角色来管理权限的模型。

**核心结构：**

```
用户 ←多对多→ 角色 ←多对多→ 权限
```

- 用户（User）：系统的使用者
- 角色（Role）：一组权限的集合，如 admin、operator、viewer
- 权限（Permission）：对特定资源的操作权，如 detection:create、training:start

**本项目实现：**

- 5 张表：`users`、`roles`、`permissions`、`user_roles`、`role_permissions`
- 三种默认角色：admin（全部权限）、operator（检测 + 训练权限）、viewer（只读权限）
- 通过中间件在 API 层校验用户权限

**RBAC 的优势：**

- 权限管理灵活：新增用户只需分配角色，无需逐一配置权限
- 支持最小权限原则：每个角色只拥有完成其职责所需的最小权限

### 3.1 Python 与后端开发

#### Q35：Python 的 asyncio 是什么？在项目中是如何使用的？

asyncio 是 Python 的异步 I/O 框架，基于事件循环（Event Loop）和协程（Coroutine）实现非阻塞并发。

**核心概念：**

- `async def`：定义协程函数
- `await`：等待异步操作完成，期间事件循环可执行其他协程
- `asyncio.run()`：启动事件循环

**本项目中的使用场景：**

1. FastAPI 路由：所有 API 处理函数使用 `async def`，支持高并发请求
2. SSE 流式对话：`async for` 迭代 LLM 的流式输出，每个 chunk 立即推送给前端
3. 数据库操作：虽然 SQLAlchemy 是同步的，但通过 `run_in_executor` 在线程池中执行
4. LLM 调用：使用 `astream_events` 异步流式调用 LLM

**asyncio vs 多线程：**

- asyncio 是协作式多任务（协程主动让出控制权），适合 I/O 密集型
- 多线程是抢占式多任务（操作系统调度），适合 CPU 密集型
- YOLO 推理是 CPU/GPU 密集型，使用后台线程而不是 async

#### Q36：FastAPI 的依赖注入是什么？在项目中怎么用的？

依赖注入（Dependency Injection）是 FastAPI 的核心特性，通过 `Depends()` 实现组件的自动创建和注入。

**本项目中的使用：**

```python
# 1. 数据库会话注入
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

@router.get("/stats")
def get_stats(db: Session = Depends(get_db)):
    ...

# 2. 当前用户认证注入
def get_current_user(token: str = Depends(OAuth2PasswordBearer(...))):
    payload = decode_access_token(token)
    user = db.query(User).filter(User.id == payload["sub"]).first()
    return user

@router.get("/me")
def get_me(user: User = Depends(get_current_user)):
    return user
```

**优势：**

- 自动管理资源生命周期（如数据库会话的创建和关闭）
- 代码解耦，便于单元测试（mock 依赖）
- 依赖链自动解析（如 `get_current_user` 依赖 `OAuth2PasswordBearer`）

#### Q37：SQLAlchemy ORM 是什么？项目中是如何使用的？

ORM（Object-Relational Mapping）将数据库表映射为 Python 类，用对象操作代替 SQL 语句。

#### Q40：前端是如何实现 SSE 流式接收的？

本项目使用 fetch + ReadableStream 实现 SSE 接收（比 EventSource 更灵活，支持 POST 请求）：

```javascript
const response = await fetch('/api/chat/stream', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ message, session_id })
})
const reader = response.body.getReader()
const decoder = new TextDecoder()

while (true) {
    const { done, value } = await reader.read()
    if (done) break
    const text = decoder.decode(value)
    const lines = text.split('\n').filter(l => l.startsWith('data:'))
    for (const line of lines) {
        const data = JSON.parse(line.slice(6))
        switch (data.type) {
            case 'thinking': showThinking(); break
            case 'text_chunk': appendText(data.content); break
            case 'tool_start': showToolCall(data.tool); break
            case 'done': finishStreaming(); break
        }
    }
}
```

**为什么不用 EventSource：**

- EventSource 只支持 GET 请求
- 本项目需要 POST 发送消息内容和文件信息
- fetch + ReadableStream 更灵活，可以自定义请求头（如 Authorization）

### 3.3 系统设计

#### Q41：如果要将这个项目部署到生产环境，你会怎么做？

**部署架构：**

```
                     ┌── Nginx（反向代理 + 静态资源 + HTTPS）
                     │        │
用户 → CDN → Nginx ──┤        ├── 前端静态文件（Vue build）
                     │        └── /api/* → Gunicorn/Uvicorn (FastAPI)
                     │                                         │
                     └── Docker Compose ───────────────────────┤
                          ├── PostgreSQL + pgvector            │
                          ├── Redis                            │
                          ├── MinIO                            │
                          └── Ollama（可选）                    │
```

**关键步骤：**

1. 前端：`npm run build` 生成静态文件，Nginx 托管
2. 后端：Uvicorn + Gunicorn 多 worker 部署，配置 `--workers 4`
3. Nginx：配置反向代理、HTTPS（Let's Encrypt）、gzip 压缩、WebSocket 升级
4. 数据库：PostgreSQL 配置连接池、定期备份
5. Redis：配置持久化（RDB + AOF）
6. MinIO：配置纠删码保证数据可靠性
7. 监控：接入 Prometheus + Grafana，配置告警规则
8. 日志：统一收集到 ELK 或 Loki
9. CI/CD：GitHub Actions 自动测试 → Docker 镜像构建 → 自动部署

#### Q42：如何保证系统的高可用性？

1. **应用层：**
   - Gunicorn 多 worker 进程，单 worker 崩溃不影响整体服务
   - 健康检查接口 `/api/health`，配合负载均衡自动剔除故障节点
2. **数据层：**
   - PostgreSQL 主从复制 + 自动故障转移
   - Redis Sentinel 哨兵模式，主节点故障自动切换
   - MinIO 纠删码，数据多副本冗余
3. **缓存策略：**
   - 模型单例缓存避免重复加载
   - Redis 缓存高频查询结果
   - 前端请求防抖避免重复提交
4. **限流与降级：**
   - 基于 Redis 的分布式限流，防止恶意请求
   - 视频检测等重计算任务异步处理
   - LLM 调用超时降级，返回友好提示
5. **监控告警：**
   - 系统指标监控（CPU、内存、磁盘）
   - 应用指标监控（QPS、响应时间、错误率）
   - 异常自动告警（邮件/钉钉/飞书）

#### Q43：如果检测请求量突然增大 10 倍，系统会怎样？你会如何优化？

**瓶颈分析：**

1. YOLO 推理：CPU 推理约 200-500ms/张，单线程吞吐量有限 → 最大瓶颈
2. 数据库连接：SQLAlchemy 连接池可能耗尽
3. MinIO 上传：I/O 带宽限制
4. LLM 调用：API 限流和延迟

**优化方案：**

短期应急：

- 增加限流阈值限制，返回 429 防止系统崩溃
- 检测任务放入消息队列（Celery/RabbitMQ），异步处理
- 增加 Uvicorn worker 数量

中期优化：

- 引入 GPU 推理，吞吐量提升 10-50 倍
- 模型量化（FP16/INT8）或使用 TensorRT/ONNX Runtime 加速
- 引入批推理（Batch Inference），多张图片一次前向传播
- 数据库读写分离，查询走从库

长期架构：

- 检测服务独立部署，支持水平扩缩容
- 引入 Kubernetes 自动伸缩
- CDN 加速图片分发
- 引入对象存储的预签名 URL，减少后端 I/O 压力

### 3.4 项目深度追问

#### Q44：对话记忆是如何实现的？

本项目使用 Redis 实现对话记忆，支持跨轮次上下文：

**实现方式：**

- 每个会话（session）在 Redis 中有独立的 key
- 每轮对话的 user message 和 AI response 追加到消息列表
- Agent 调用 LLM 时，将历史消息作为上下文传入

**单 Agent 模式：**

- 使用 LangChain 的 ConversationBufferMemory 或自定义 Redis Memory
- 每次调用 LLM 时携带完整对话历史

**多 Agent 模式：**

- AgentState 中的 messages 字段使用 `add_messages` reducer 自动追加
- 所有 Agent 共享同一个消息列表，可看到之前的对话上下文

**Redis 的优势：**

- 读写速度极快（内存存储）
- 支持 TTL 过期自动清理旧会话
- 分布式部署时多个后端实例共享记忆

#### Q45：MinIO 在项目中承担什么角色？为什么不用本地文件系统？

**MinIO 承担的角色：**

- 存储检测原图和标注图
- 存储视频检测的标注视频
- 存储训练好的模型权重
- 存储用户上传的文件

**不用本地文件系统的原因：**

1. 可扩展性：MinIO 兼容 S3 API，未来可以无缝迁移到 AWS S3、阿里云 OSS
2. 分布式：多实例部署时共享存储，本地文件系统无法共享
3. 高可用：纠删码机制保证数据可靠性
4. 管理功能：内置 Web Console、预签名 URL、生命周期管理
5. 标准化：统一的对象存储 API，便于前后端直接访问（预签名 URL）

#### Q46：项目中如何处理错误和异常？

本项目实现了多层次的异常处理机制：

1. **全局异常处理器（FastAPI Exception Handler）：**

   ```python
   @app.exception_handler(Exception)
   async def global_exception_handler(request, exc):
       logger.error("系统异常: %s", str(exc), exc_info=True)
       return JSONResponse(
           status_code=500,
           content={"error": "系统内部错误", "detail": "请稍后重试"}
       )
   ```

   - 捕获所有未处理的异常，防止暴露内部堆栈信息

2. **业务异常（自定义 Exception）：**
   - AuthenticationError：认证失败，返回 401
   - PermissionDenied：权限不足，返回 403
   - ResourceNotFound：资源不存在，返回 404
3. **Service 层 try-except：**
   - 每个 Service 方法内部捕获异常，记录日志，返回错误信息
   - 如检测服务异常时返回 `{"error": "检测失败: ..."}`
4. **数据库事务：**
   - 使用 try-finally 确保 `db.close()` 释放连接
   - 异常时自动回滚事务
5. **前端处理：**
   - Axios 拦截器统一处理 HTTP 错误码
   - Token 过期自动跳转登录页

#### Q47：项目中如何保证前后端的数据一致性？

1. **Schema 验证：**
   - 后端使用 Pydantic Schema 定义请求/响应模型，自动验证数据类型和格式
   - 前端使用 TypeScript 类型定义（或 JSDoc）保持接口类型一致
2. **统一的数据格式：**
   - 所有 API 返回统一的 JSON 格式
   - 错误响应统一为 `{"error": "...", "detail": "..."}`
3. **数据库事务：**
   - 检测任务创建 → 结果保存 → 提交事务，保证原子性
   - 异常时回滚，避免脏数据
4. **乐观锁/版本号：**
   - 模型版本管理中，`is_default` 字段保证只有一个默认模型
5. **状态同步：**
   - 异步任务（视频检测、模型训练）通过状态轮询或 SSE 推送进度
   - 前端根据任务状态（processing/completed/failed）更新 UI

#### Q50：你对 AI Agent 的未来发展有什么看法？

1. 多模态 Agent：从纯文本交互发展为支持图像、视频、语音的多模态理解和生成

3. 自主规划：Agent 不仅能调用工具，还能自主分解复杂任务、制定执行计划
