## 第一部分：项目答辩高频提问

### 1.1 项目概述类

#### Q1：请简要介绍一下你的项目

本项目是「基于 YOLO11n 与多智能体架构的织物表面缺陷检测平台」，是一个将计算机视觉（CV）与大语言模型（LLM）深度融合的全栈 AI 应用。

**核心能力包括：**

- 缺陷检测：支持单图、批量、ZIP 压缩包、视频、摄像头（浏览器帧流 / IP 摄像头拉流）五种模式，基于 YOLO11n 模型检测织物缺陷（如划痕 scratch、破洞 hole、污渍 stain、纱线缺陷 yarn_defect）
- 智能对话：基于 LangChain/LangGraph 构建的多智能体系统，Supervisor 将一条消息规划为多个子任务，检测/分析/问答三个专家 Agent 支持并行与依赖递进执行
- RAG 知识库：pgvector 向量检索 + 离线词法检索双通道，支持织物缺陷领域知识问答，可上传 md/txt/pdf 文档并后台自动重建索引
- 模型训练：YOLO11 系列（n/s/m/l/x）训练、评估、导出、版本管理全流程
- 数据看板：检测数据的统计分析与可视化，支持自定义日期区间和按缺陷类别过滤

**技术栈**：后端 Python 3.10 + FastAPI + SQLAlchemy + Alembic，前端 Vue 3（`<script setup>`）+ Vite + Element Plus + Pinia + ECharts，数据库 PostgreSQL 15 + pgvector，缓存 Redis 7，对象存储 MinIO，AI 框架 LangChain + LangGraph + Ultralytics YOLO。

#### Q2：你为什么选择这个课题？项目的创新点在哪里？

**选题原因：**

- 织物缺陷检测是纺织行业质检环节的核心痛点，人工验布效率低、漏检率高
- 传统检测系统操作门槛高，将 AI 检测能力与自然语言交互结合，可以显著降低质检人员的使用门槛

**创新点：**

1. CV + LLM 深度融合：通过 Agent 工具调用机制，让大模型自主决定何时调用检测工具，实现"对话即检测"——用户上传图片说"检测一下"，Agent 自动执行 YOLO 推理并解读结果
2. 多智能体规划-并行架构：Supervisor 用 LLM 把一条复合请求（如"检测这张图片，并告诉我什么是 YOLO"）规划成带依赖关系（depends_on）的子任务 DAG，多个专家 Agent 并行执行、按序合并流式输出；上游检测结果的结构化数据还能注入下游分析任务
3. RAG 知识库双通道：pgvector 向量检索为主，pgvector/Embedding 不可用时自动降级为本地词法检索（永远可离线运行），并向用户如实标注 `retrieval_mode=lexical_fallback`
4. 五种检测模式一体化 + 模型版本全生命周期管理（训练 → 评估 → 导出 → 设为全局默认模型）

#### Q3：项目的技术架构是怎样的？

```
前端 (Vue 3) ──HTTP(SSE)/WebSocket──▶ FastAPI 后端 ──▶ PostgreSQL 15 + pgvector
                                          │
                                          ├──▶ Redis（对话记忆/限流/视频任务进度，含内存降级）
                                          ├──▶ MinIO（原图/标注图/标注视频/模型权重）
                                          ├──▶ YOLO 模型（推理/训练，Ultralytics）
                                          └──▶ LLM API（通义千问 qwen-plus，OpenAI 兼容）
```

**后端采用分层架构：**

- API 层：FastAPI 路由（auth/training/chat/detection/dashboard/history/user/knowledge + health）
- Agent 层：LangGraph Supervisor 规划 + asyncio 并行编排三个专家 Agent
- Service 层：检测、看板、历史、用户等业务服务；训练模块独立在 `app/training/`
- Entity 层：SQLAlchemy ORM 模型（16 张表）+ Pydantic Schema
- Storage 层：Redis、MinIO、pgvector 客户端封装

全局中间件：CORS → 请求日志 → Redis 限流。

### 1.2 核心技术实现类

#### Q4：多智能体架构是如何实现的？

核心流程：

```
用户输入 → Supervisor（LLM 规划子任务 JSON）→ Detection/Analysis/QA 专家（并行 + 依赖递进）→ 合并流式输出
```

**具体实现（`app/agent/`）：**

1. 规划：`SupervisorAgent.route()` 用 LLM 按 `SUPERVISOR_PLANNING_PROMPT` 输出子任务数组，每项含 `{"agent", "task", "depends_on"}`；同一专家的多条子任务会合并（每个专家最多执行一次）。LLM 不可用时三级降级：JSON 规划 → 单标签正则解析 → 关键词规则（关键词兜底故意只路由单个专家）
2. 依赖清洗：`sanitize_dependencies()` 丢弃自引用/悬空引用，用 Kahn 拓扑算法检测环，成环则清空全部依赖降级为纯并行——保证不会死锁
3. 共享状态 `AgentState`（TypedDict）：`messages` 是唯一带 `add_messages` reducer 的字段；三个专家各写各的 `detection_result/analysis_result/qa_result`，因为 LangGraph 的 LastValue 通道不允许同一 superstep 两个节点写同一 key（否则抛 `InvalidUpdateError`）
4. 并行执行在编排器（`MultiAgentOrchestrator._parallel_chat_stream`）的 asyncio 层完成：每个专家一个 `asyncio.create_task`，事件经共享 `asyncio.Queue` 合并；有依赖的专家先 `await` 上游的 `asyncio.Event`，上游的检测结构化数据（total_objects/detected_classes/class_counts）以 `[DEPENDENCY_DATA]` JSON 注入下游任务提示词
5. 输出合并：`tool_call/tool_result` 事件带 agent 标签实时透传；文本按计划顺序"有序渐进冲刷"——首位专家实时直出，其余先缓冲，前序完成后依序输出分节标题再冲刷

一个值得说明的设计点：LangGraph 状态图在运行时只负责**规划路由**（专家节点是打标桩函数），真正的专家执行放在编排器 asyncio 层，这样才能实现依赖等待和流式合并这类图内不好表达的控制流。

#### Q5：一条消息是怎么决定由一个还是多个 Agent 处理的？

系统没有显式的"单 Agent / 多 Agent 模式"开关，只有一个统一入口 `MultiAgentOrchestrator.chat_stream`，由 Supervisor 的规划结果自动分流：

| 情况 | 执行路径 |
| --- | --- |
| plan 只有 1 个子任务 | 走单专家路径，行为与传统单 Agent 一致，专家自行管理记忆和附件 |
| plan 有多个子任务 | 走 `_parallel_chat_stream` 并行路径，专家均以 `record_memory=False` 执行，编排器最后统一写一次对话记忆 |

每个专家本质上是一个 LangChain OpenAI-tools Agent：`create_openai_tools_agent(llm, tools, prompt)` + `AgentExecutor(max_iterations=8, return_intermediate_steps=True)`，流式用 `astream_events(version="v2")`。

**职责分离的好处**：每个专家只绑定领域内工具（detection 5 个 / analysis 4 个 / qa 1 个，而不是全部 10 个），提示词更聚焦，LLM 选工具准确率更高。analysis 专家还配置了"强制先调工具"校验（`required_tool_names` + 1 次重试），防止它凭空编造统计数字。

**失败处理**：单个专家失败降级为该分节内的 ⚠️ 提示文本；只有全部专家都失败才对外发 `error` 事件。

#### Q6：RAG 知识库是怎么实现的？

RAG 分为索引构建和检索问答两个阶段（`app/rag/`）：

**索引构建：**

1. 文档加载：读取 `backend/knowledge_base/` 下的 `.md/.txt/.pdf` 文件（PDF 用 pypdf 抽取文本，失败跳过不中断）。内置三篇文档：织物缺陷知识（fabric_defects.md）、YOLO 基础（yolo_basics.md）、模型评估指标（model_evaluation.md）
2. 文本分块：先按 Markdown 标题正则切成 section（标题行保留在块内），再在 section 内滑窗切块——500 字符/块，50 字符重叠（步长 450）
3. 向量化：默认 `text-embedding-v3`（1024 维），走 DashScope 的 OpenAI 兼容接口，每批 10 条（DashScope 单次上限）
4. 向量存储：`replace()` 全量替换写入 pgvector 表 `knowledge_embeddings`

**检索问答：**

1. 查询向量化 → pgvector 余弦距离 `<=>` 检索，Top-K 默认 3，返回 `score = 1 - cosine_distance`
2. 降级机制：向量表为空或 Embedding/pgvector 异常时，自动退回内存词法检索（英文按 `[a-z0-9_]+` 分词 + 中文二字 bigram，交集打分，整句子串命中加 5 分），并返回 `mode="lexical_fallback"` 和降级原因——QA Agent 必须向用户如实说明当前是降级检索
3. 检索结果拼接为上下文 → LLM 生成回答

**工程细节**：文件增删后 `schedule_rebuild()` 在单个后台守护线程重建索引，用挂起标记去抖（连续增删只在末尾补跑一轮）；只有默认 `knowledge_base` 目录支持向量索引。

#### Q7：SSE 流式对话是怎么实现的？

`POST /api/chat/stream` 返回 `StreamingResponse(media_type="text/event-stream")`，每条事件为 `data: {JSON}\n\n`。

**事件类型（以代码为准）：**

| 事件 | 含义 |
| --- | --- |
| `session` | 回传/生成的 session_id |
| `thinking` | 开始分析（chat.py 发一次固定文案） |
| `agent_route` | 路由结果：单专家含 `agent`，并行含 `agents` 和 `plan` |
| `text_chunk` | LLM 增量文本（来自 `on_chat_model_stream`） |
| `tool_call` / `tool_result` | 工具开始/结束（来自 `on_tool_start`/`on_tool_end`），并行时带 agent 标签 |
| `error` | 全部专家失败时的顶层错误 |
| `done` | 结束，随后再发 `data: [DONE]` 兼容旧客户端 |

后端事件源是 LangChain 的 `astream_events(version="v2")`；响应头带 `X-Accel-Buffering: no` 防止反向代理缓冲。流结束后把 assistant 消息落库（tool_result 落库前剥离 base64）。

**前端**用 fetch + ReadableStream 逐行解析 `data:` 前缀（而不是 EventSource，因为需要 POST 带 body 和 Authorization 头），按事件类型驱动 UI：思考动画、逐字输出、工具链面板等。

#### Q8：视频检测是如何实现的？

核心思路是**动态帧采样 + 异步后台处理 + 浏览器兼容转码**（`detection_service.py`）：

1. 上传校验：限 50MB，支持 mp4/avi/mov/mkv/wmv/flv
2. 帧采样：默认 `frame_sample_rate=5`（每 5 帧取 1）、`max_frames=50`（上限 500）；实际采样间隔 `effective_interval = max(frame_sample_rate, total_frames // max_frames)`，保证长视频也均匀覆盖且推理帧数有上限
3. 推理与绘制：只对采样帧执行 YOLO 推理；非采样帧复用上一次的检测框绘制，输出视频每帧都有标注但推理量大幅下降
4. 异步执行：后台守护线程处理，进度写 Redis `video_task:{id}`（1 小时过期），前端轮询 `GET /api/detection/video/status/{task_id}`
5. 转码：OpenCV 先输出 mp4v，再用 ffmpeg 转 H.264（libx264、crf 23、yuv420p、+faststart，超时 300 秒），失败则回退原始文件——保证浏览器可直接播放
6. 结果持久化：标注视频上传 MinIO `detections/{task_id}/annotated_video.mp4`，关键帧只保留前 6 帧的 base64 缩略图，检测明细写 PostgreSQL

另一个一致性设计：视频任务在提交时锁定 `model_version_id`，处理中途管理员切换全局模型不会影响进行中的任务。

#### Q9：摄像头实时检测是如何实现的？

基于 WebSocket（`/api/detection/camera`）双向通信，支持两种视频来源：

1. `browser` 模式：前端 getUserMedia 抓帧，编码 base64 经 WebSocket 发送，后端解码 → YOLO 推理 → 返回标注图 base64
2. `ip_webcam` 模式：后端直接拉取 IP 摄像头流；出于 SSRF 防护，URL 必须是局域网私有 IP 且协议为 http/https

**关键设计：**

- 鉴权：WebSocket 无法带标准 Authorization 头，首条 config 消息携带 `access_token`，校验失败以自定义 close code 4401/4403 关闭
- 性能：CPU 模式下 imgsz 降到 416（GPU 640），JPEG 质量降到 70，模型单例缓存避免重复加载
- 前端控制抓帧频率，支持实时调整置信度和 IoU 阈值

#### Q10：模型训练功能是如何实现的？

基于 Ultralytics YOLO11（`app/training/training_service.py`）：

1. 任务创建：选择模型规格（yolo11 n/s/m/l/x，默认 n）、配置超参数（默认 epochs=50、img_size=640、batch_size=8、optimizer=SGD、lr0=0.01，支持数据增强配置）
2. 数据准备：`dataset_splitter.py` 自动划分 train/val/test = 8:1:1
3. 异步执行：训练在后台守护线程运行；评估等模型重任务放到 `ProcessPoolExecutor(spawn)` 独立进程——避免 CPU 密集推理占住 GIL 饿死 uvicorn 事件循环
4. 指标监控：解析 Ultralytics `results.csv`，逐 epoch 把 box_loss/cls_loss/dfl_loss/mAP/precision/recall 写入 `training_metrics` 表；前端训练页每 5 秒轮询
5. 评估：验证集/测试集评估结果写 `model_evaluations` 表，含 per-class AP
6. 导出与版本管理：把 `runs/train/{uuid}/weights/best.pt` 复制到 `backend/models/{scene}_{version}/`，登记 `model_versions` 表，可上传 MinIO 并设为**全局默认模型**（检测服务只加载 `is_global_default=True` 的模型）
7. 历史恢复：应用启动时 `rescan_tasks()` 扫描 `runs/train` 目录恢复训练历史，也可手动 `POST /api/training/rescan`

### 1.3 数据库设计类

#### Q11：数据库是如何设计的？

SQLAlchemy 模型共 **16 张表**（`app/entity/db_models.py`），分为五大模块：

**用户权限模块（RBAC + 会话）：**

- `users`、`auth_sessions`（刷新令牌会话）、`roles`、`permissions`、`user_roles`、`role_permissions`
- 三种默认角色：`quality_inspector`（质检员）、`production_manager`（生产管理）、`system_admin`（系统管理员）

**检测业务模块：**

- `detection_scenes`：检测场景（类别列表、中文名映射 `class_names_cn`，如 {"hole": "破洞"}）
- `detection_tasks`：检测任务（single/batch/video/camera）
- `detection_results`：单条检测结果（bbox、置信度、类别及中文名、标注图）

**模型管理模块：**

- `training_tasks`、`training_metrics`（逐 epoch 指标）、`model_evaluations`（评估结果）、`model_versions`（版本 + `is_global_default` 全局默认标记 + MinIO 路径）

**智能体模块：**

- `chat_sessions`（session_uuid、title、message_count 等）
- `chat_messages`（role、content、`agent_used`（并行时逗号连接如 `detection,qa`）、tool_calls JSON、tool_result（落库前剥 base64）、attachments（只存 MinIO object_name，读历史时实时重签 URL）、tokens_used、latency_ms）

**系统运维模块：**

- `operation_logs`：操作审计日志

**向量存储**：`knowledge_embeddings` 表不在 ORM 中，由 pgvector 客户端用原生 SQL 建表（content TEXT、metadata JSONB、embedding vector(1024)）。

Schema 变更统一走 Alembic 迁移。

### 1.4 安全与性能类

#### Q12：项目中的安全机制有哪些？

1. 认证：JWT（HS256），access token 30 分钟过期、refresh token 35 分钟；payload 含 exp/iat/jti/type，access 与 refresh 用 type 字段区分互斥；另有 7 天有效的文件访问令牌用于媒体资源签名
2. 密码安全：bcrypt 加密，hash 和 verify 都把 UTF-8 编码超 72 字节的密码显式截断到 72 字节（bcrypt 的固有限制，显式处理避免不一致）
3. RBAC：用户-角色-权限三级模型；不仅在 API 层校验，**Agent 工具执行前也做权限检查**（检测工具需 DETECTION_EXECUTE、统计需 DASHBOARD_READ_ANY、知识检索需 KNOWLEDGE_READ 等），防止通过对话绕过权限
4. API 限流：基于 Redis 的固定窗口限流中间件，按路径差异化限额
5. 上传安全：聊天附件只能引用 `/api/chat/upload` 上传到受控临时目录的文件（路径校验防目录穿越）；类型/大小/数量白名单校验
6. SSRF 防护：IP 摄像头 URL 强制局域网私有 IP + http/https
7. WebSocket 鉴权：首条消息校验 token，失败用 4401/4403 关闭
8. 全局异常处理不暴露堆栈；`operation_logs` 记录关键操作审计

#### Q13：限流中间件是怎么实现的？

基于 Redis 的**固定窗口计数**算法（`app/middleware/rate_limiter.py`）：

1. 中间件拦截请求，取客户端 IP（顺序：`X-Forwarded-For` 第一段 → `X-Real-IP` → `request.client.host`，兼容反向代理）
2. 以 `rate_limit:{path}:{ip}` 为 key，调用 `RedisClient.increment_with_window(key, window)` 原子计数——**只在 key 首次创建时设置过期时间**。这是一个踩过坑的点：早期实现每次请求都重置过期时间，变成"滑动过期"，被持续轮询的接口会永久限流
3. 超限返回 429 Too Many Requests
4. Redis 异常时放行（可用性优先）；`/docs`、`/redoc` 跳过限流

**限流策略：**

- 默认：60 次/60 秒
- 单图检测 30 次/分钟、批量 10 次/分钟、视频 5 次/分钟、摄像头 10 次/分钟

Redis 不可用时自动降级为进程内内存计数（开发可用，多进程部署下不共享）。

### 1.5 项目难点与优化类

#### Q14：项目中遇到了哪些技术难点？如何解决的？

**难点 1：多专家并行时的流式输出合并**

- 问题：多个专家同时产生 text_chunk，直接混流用户不可读；但完全串行又浪费时间
- 解决：工具事件实时透传（带 agent 标签），文本采用"有序渐进冲刷"——首位专家实时直出，其余缓冲，前序完成后按计划顺序发分节标题再冲刷；单专家失败降级为节内 ⚠️ 提示而不是整体报错

**难点 2：子任务之间存在数据依赖**

- 问题："检测这张图并分析结果"——分析必须等检测完成且要拿到检测数据
- 解决：Supervisor 输出 depends_on，编排器用 asyncio.Event 做依赖等待；上游检测的结构化产物（目标数/类别统计）以 `[DEPENDENCY_DATA]` JSON 注入下游提示词；依赖清洗（去自引用/悬空引用，Kahn 检环，成环降级纯并行）保证不死锁

**难点 3：工具结果的 token 开销与前端展示冲突**

- 问题：检测工具返回 base64 标注图，直接回传 LLM 会浪费大量 token 甚至超上下文
- 解决：双通道设计——给 LLM 的返回值剥离 base64；完整结果存入 contextvar，`on_tool_end` 时取出发给前端。因为 LangChain 在子线程执行同步工具、`ContextVar.set()` 不回传父上下文，实际用可变 dict holder 传递

**难点 4：CPU 密集任务阻塞异步服务**

- 问题：模型评估等重计算在 FastAPI 进程内跑会占住 GIL，导致接口卡死
- 解决：训练用后台线程；评估用 `ProcessPoolExecutor(spawn)` 独立进程；视频检测后台线程 + Redis 进度轮询

**难点 5：外部依赖不可用时的可用性**

- 解决：全链路优雅降级——Redis 挂了退内存存储、限流放行；pgvector/Embedding 挂了退词法检索并向用户标注 `lexical_fallback`；Supervisor 的 LLM 挂了退正则/关键词路由；ffmpeg 转码失败退原始编码视频

**难点 6：per-request 上下文注入工具**

- 问题：LangChain `@tool` 是模块级函数，用户身份不能作为 LLM 参数传递（LLM 可能篡改 user_id 越权）
- 解决：user_id、scene_id、session_id、附件名映射全部通过 contextvars 注入，工具内部读取，LLM 完全接触不到身份参数

#### Q15：项目还有哪些可以优化的地方？

1. Agent：目前 analysis 专家已有"强制先调工具"机制，可进一步扩展为通用的反幻觉校验；引入计划可视化让用户确认/修改子任务
2. RAG：pgvector 表目前是顺序扫描，数据量大后应加 HNSW/IVFFlat 索引；引入 Hybrid Search（现有词法通道与向量通道融合打分）和 Reranker；分块可保留标题层级 metadata 便于溯源
3. 检测性能：引入 GPU 推理、TensorRT/ONNX Runtime、模型量化（FP16/INT8）、批推理
4. 视频检测：引入目标跟踪（ByteTrack/DeepSORT）实现跨帧目标 ID 一致与精确计数；可加场景变化检测跳过静止画面
5. 限流：固定窗口存在窗口边界突刺问题，可升级为滑动窗口或令牌桶
6. 记忆：Redis 对话记忆目前保留最近 20 条、TTL 24 小时，可加长期记忆摘要
7. 可观测性：接入 Prometheus + Grafana、分布式链路追踪

## 第二部分：考研复试面试高频题

### 2.1 计算机视觉基础

#### Q16：什么是目标检测？与图像分类有什么区别？

**图像分类**：判断一张图片属于哪个类别（整图一个标签）。

**目标检测**：不仅识别图中有什么目标，还要用边界框（Bounding Box）定位目标位置，即同时完成分类 + 定位。输出格式通常为 `(class, confidence, x1, y1, x2, y2)`。

**目标检测的难点：**

- 目标尺度变化（织物上大面积污渍与细小纱线缺陷共存）
- 低对比度目标（缺陷与织物纹理背景相似）
- 背景纹理复杂、周期性图案干扰
- 实时性要求（在线验布）

本项目使用 YOLO11n 检测织物表面缺陷，主力数据集为四类：划痕（scratch）、破洞（hole）、污渍（stain）、纱线缺陷（yarn_defect）；同时准备了 2 类、3 类、21 类等多个数据集变体用于对比实验。

#### Q17：什么是 YOLO？YOLO 的核心思想是什么？

YOLO（You Only Look Once）是单阶段（One-Stage）目标检测算法，核心思想是把检测转化为回归问题。

**核心思想：**

1. 将输入图像划分为网格
2. 每个位置直接预测边界框和类别概率
3. 一次前向传播得到所有目标

**与 Two-Stage 方法（如 Faster R-CNN）的对比：**

| 维度 | YOLO (One-Stage) | Faster R-CNN (Two-Stage) |
| --- | --- | --- |
| 流程 | 直接回归 bbox + 类别 | 先提候选框，再分类回归 |
| 速度 | 快，适合实时检测 | 慢，精度较高 |
| 小目标 | 相对较弱 | 相对较好 |

**YOLO11 的特点：**

- 更高效的骨干网络（Backbone）与改进的特征金字塔（Neck）
- 解耦头分别预测分类和回归
- Anchor-Free 设计，减少超参数
- n/s/m/l/x 五档规格，本项目选 n（nano）兼顾精度与 CPU 推理速度

#### Q18：什么是 IoU？什么是 NMS？

**IoU（交并比）** = 预测框与真实框交集面积 / 并集面积，取值 [0, 1]，越大定位越准；既用于评估，也用于 NMS 去重。

**NMS（非极大值抑制）**：去除冗余重叠框。流程：

1. 按置信度降序排列
2. 取最高分框保留
3. 去除与它 IoU 超过阈值的其余框
4. 重复直至处理完毕

本项目 NMS IoU 阈值默认 0.45，置信度阈值默认 0.25，两者都可在检测请求中调整。

#### Q19：什么是 mAP？如何计算？

mAP（mean Average Precision）是目标检测最常用的评估指标：

1. Precision = TP / (TP + FP)；Recall = TP / (TP + FN)
2. 以 Recall 为横轴、Precision 为纵轴绘制 P-R 曲线
3. AP = P-R 曲线下面积，衡量单个类别精度
4. mAP = 所有类别 AP 的平均值

**常用变体**：mAP@0.5（VOC 标准）、mAP@0.5:0.95（COCO 标准，更严格）。

本项目训练时逐 epoch 记录 box_loss、cls_loss、dfl_loss、mAP50、mAP50-95、precision、recall 到 `training_metrics` 表；评估阶段还输出 per-class AP（例如可以看出"污渍"类因边界模糊 AP 通常低于"破洞"类）。

#### Q20：什么是混淆矩阵？精确率和召回率有什么关系？

**混淆矩阵：**

|  | 预测为正 | 预测为负 |
| --- | --- | --- |
| 实际为正 | TP | FN |
| 实际为负 | FP | TN |

Precision = TP / (TP + FP)：报出的缺陷里多少是真缺陷（误检率的反面）
Recall = TP / (TP + FN)：真实缺陷里多少被找到（漏检率的反面）

**两者通常矛盾**：提高置信度阈值 → Precision 升、Recall 降（漏检增多）；反之误检增多。F1 = 2PR/(P+R) 是调和平均。

在织物质检场景里，漏检（把有缺陷的布当合格品放行）代价通常高于误检，所以实际部署可以适当调低置信度阈值偏向高 Recall；本项目默认 0.25，检测接口支持按需调整。

### 2.2 深度学习与模型训练

#### Q21：什么是过拟合？如何防止过拟合？

**过拟合**：训练集表现好、验证/测试集表现差，模型"记住"了数据而非学到规律。

**防止方法：**

1. 数据增强：翻转、旋转、缩放、颜色抖动（Ultralytics 内置 Mosaic、MixUp；本项目训练接口支持 augment_config 自定义）
2. 正则化：L1/L2、Dropout
3. 早停：验证 loss 不再下降时停止
4. 增加数据量
5. 减小模型复杂度（选 yolo11n 而非更大规格）
6. 学习率调度：WarmUp、余弦退火

本项目通过训练页的逐 epoch 指标曲线（train loss 降但 val 指标不升即过拟合信号）判断，并可停止任务调参重训。

#### Q22：YOLO 训练中的 loss 有哪些？分别代表什么？

1. box_loss：边界框回归损失（CIoU），衡量定位偏差
2. cls_loss：分类损失（BCE），衡量类别预测
3. dfl_loss：分布焦点损失，把框回归建模为概率分布，提高模糊边界（如污渍边缘）的定位精度

三者应随 epoch 递减趋稳；train loss 降而 val loss 升说明过拟合；mAP 是最综合的指标。这些指标在本项目中逐 epoch 解析 `results.csv` 持久化，前端 ECharts 实时绘制曲线。

#### Q23：什么是迁移学习？在目标检测中如何应用？

**迁移学习**：把一个任务上学到的知识迁移到相关任务，减少对大量标注数据的依赖。

**应用方式：**

1. 预训练权重初始化：本项目训练即从 COCO 预训练的 `yolo11n.pt` 出发微调
2. Fine-tuning：冻结 Backbone 只训 Head，或全层小学习率微调
3. 场景适配：织物图像与日常图像域差异大（纹理背景、低对比度目标），必须用织物缺陷数据集微调

本项目的模型管理与之配套：训练产出登记为 `model_versions`，管理员将某个版本设为全局默认（`is_global_default=True`），检测服务只加载该模型——保证线上模型是经过评估挑选的微调版本，而不是随意回退到通用预训练权重。

### 2.3 大语言模型与 Agent

#### Q24：什么是 LangChain？什么是 LangGraph？

**LangChain**：构建 LLM 应用的框架，核心抽象：

- Model I/O：统一 LLM 调用接口（OpenAI 兼容，本项目接通义千问）
- Tools：让 LLM 调用外部函数（`@tool` 装饰器）
- Agent：LLM 自主决定调用哪些工具（本项目用 `create_openai_tools_agent` + `AgentExecutor`）
- Memory：对话记忆

**LangGraph**：LangChain 团队的多 Agent 编排框架，基于状态图：

- Node（节点）/ Edge（边）/ Conditional Edge（条件边，`add_conditional_edges` 返回列表即可并行扇出）
- State：TypedDict 共享状态，可为字段配 reducer（如 `add_messages` 自动追加消息）

本项目的图结构是 supervisor → detection/analysis/qa（可并行）→ summarize → END；实践中图用于规划路由，专家的实际执行放在 asyncio 编排层以支持依赖等待和流式合并。

#### Q25：什么是 RAG？为什么需要 RAG？

RAG（检索增强生成）= 信息检索 + LLM 生成。

**为什么需要：**

1. 减少幻觉：基于真实文档回答
2. 知识时效性：可检索最新文档
3. 领域专业性：注入织物缺陷、YOLO 评估等垂直知识
4. 可溯源：答案可追溯到文档片段（本项目前端引用面板可展开/收起检索来源）

**基本流程：**

```
索引：文档 → 分块 → 向量化 → 存入向量库
问答：查询 → 向量化 → 相似度检索 → 拼接上下文 → LLM 生成
```

本项目将织物缺陷知识、YOLO 基础、评估指标三类文档向量化入 pgvector，QA 专家通过 `search_knowledge` 工具检索；还支持用户上传 md/txt/pdf 扩充知识库并后台自动重建索引。

#### Q26：什么是向量数据库？pgvector 和专用向量数据库（如 Milvus）有什么区别？

**向量数据库**：存储高维向量并支持相似度检索（余弦、欧氏等）。

| 维度 | pgvector | Milvus/Pinecone |
| --- | --- | --- |
| 部署复杂度 | 低，PostgreSQL 扩展 | 高，独立服务 |
| 适用规模 | 百万级以下 | 十亿级以上 |
| 查询能力 | SQL + 向量混合查询 | 纯向量为主 |
| 事务 | 完整 ACID | 有限 |
| 运维成本 | 低（复用现有库） | 高 |

**本项目选 pgvector 的原因**：知识库规模小（几十到几百个 chunk），复用已有 PostgreSQL 实例，SQL 管理 metadata（JSONB 列）方便。当前数据量下甚至没建向量索引（顺序扫描足够快）；规模上去后可加 HNSW 索引，这也是我预留的优化点。

#### Q27：什么是 Embedding？文本向量化的过程是怎样的？

**Embedding**：把文本映射为固定维度稠密向量，语义相近的文本在向量空间中距离相近。

**过程**：分词 → Token 向量 → Transformer 编码 → Pooling → 固定维度文本向量。

**本项目实现**（`app/rag/embedding.py`）：

- 默认模型 `text-embedding-v3`（1024 维），通过 DashScope 的 OpenAI 兼容接口调用；凭据优先级 EMBEDDING_* 独立配置 > QWEN_* > OPENAI_*
- 批量向量化每批 10 条（DashScope 单次上限）；返回维度与配置的 EMBEDDING_DIM 不一致时立即报错（防止索引与查询维度错位）
- 相似度用余弦距离（pgvector `<=>` 操作符），只衡量方向、不受向量长度影响

#### Q28：Agent 的 Tool Calling 机制是如何工作的？

1. 工具定义：`@tool` 装饰器 + docstring/类型注解生成 JSON Schema

   ```python
   @tool
   def detect_single_image(filename: str = "") -> str:
       """对会话中已上传的织物图片执行缺陷检测"""
       ...
   ```

2. 工具绑定：`create_openai_tools_agent(llm, tools, prompt)`，每个专家只绑定领域工具（detection 5 个检测工具、analysis 4 个统计查询工具、qa 1 个知识检索工具）
3. LLM 决策：分析意图，决定是否调用、调哪个、传什么参数
4. 框架执行工具，结果回传 LLM 生成回复（AgentExecutor 最多迭代 8 轮）

**本项目的两个关键设计：**

- 身份不走 LLM 参数：user_id/session_id 等通过 contextvars 注入工具，LLM 无法伪造身份越权；工具执行前还做 RBAC 权限检查
- 双通道返回：给 LLM 的结果剥离 base64 标注图（省 token），完整结果经 contextvar 在 `on_tool_end` 发给前端展示

### 2.4 软件工程与系统设计

#### Q29：什么是 RESTful API？你的项目是如何设计 API 的？

RESTful 核心原则：资源导向（URL 名词 + HTTP 方法动词）、无状态、统一接口。

**本项目的 API 设计：**

- 前缀 `/api/{auth,training,chat,detection,dashboard,history,user,knowledge}` + 无前缀 health
- FastAPI 自动生成 Swagger（`/docs`）/ReDoc；Pydantic Schema 做请求/响应验证
- 流式对话用 SSE（`POST /api/chat/stream`），摄像头实时检测用 WebSocket（`/api/detection/camera`）
- 异步长任务（视频检测、训练）返回 task_id，前端轮询 status 接口

**示例：**

- `POST /api/detection/single` — 单图检测
- `GET /api/detection/video/status/{task_id}` — 视频任务进度
- `POST /api/knowledge/rebuild` — 202 异步重建知识索引

#### Q30：什么是 JWT？JWT 的工作原理是什么？

JWT（RFC 7519）是自包含的 Token 标准，结构 `Header.Payload.Signature`（HS256 签名）。

**流程**：登录验证密码 → 签发 JWT → 前端后续请求带 `Authorization: Bearer <token>` → 服务器验签 + 验过期 → 提取用户身份。

**优缺点**：无状态、易分布式；但签发后难主动撤销、Payload 只是 Base64 编码非加密。

**本项目实现**（`app/core/security.py`）：

- HS256；access token 30 分钟、refresh token 35 分钟，payload 含 exp/iat/jti（uuid4）/type，用 type 字段严格区分两种 token 防止互相冒用
- 刷新会话记录在 `auth_sessions` 表，支持会话管理
- 另有 7 天有效的文件访问令牌，用于图片/视频等媒体资源的签名 URL（避免长连接场景频繁 401）

#### Q31：什么是 RBAC 权限模型？

RBAC（基于角色的访问控制）：`用户 ←多对多→ 角色 ←多对多→ 权限`。

**本项目实现：**

- 4 张关系表：`roles`、`permissions`、`user_roles`、`role_permissions`
- 三种默认角色（贴合织物质检业务）：
  - `quality_inspector` 质检员：对话、知识库、检测、查看自己的历史
  - `production_manager` 生产管理：额外可查看所有人历史与看板统计
  - `system_admin` 系统管理员：全部权限（含用户管理、训练、模型管理）
- 权限校验有两层：API 依赖注入校验 + Agent 工具执行前校验（防止通过对话入口绕过 API 层权限）

**优势**：新增用户只需分配角色；每个角色遵循最小权限原则。

### 2.5 项目架构深入追问

#### Q32：为什么选择 FastAPI 而不是 Flask/Django？

| 维度 | FastAPI | Flask | Django |
| --- | --- | --- | --- |
| 异步支持 | 原生 async/await | 需额外配置 | 部分支持 |
| 性能 | 高（Starlette + Uvicorn） | 中 | 中 |
| 类型验证 | Pydantic 自动验证 | 手动/第三方 | Form/Serializer |
| API 文档 | 自动 Swagger + ReDoc | 手动 | 需 DRF |

**选择原因：**

1. 原生异步：SSE 流式对话（astream_events 异步迭代）、WebSocket 摄像头检测都依赖 async
2. Pydantic 2 集成：请求/响应自动验证
3. 自动文档方便前后端联调
4. ASGI 高性能

同时也清楚它的边界：CPU 密集的 YOLO 推理/评估不能放在事件循环里，所以配了后台线程和 ProcessPoolExecutor。

#### Q33：项目中用了哪些设计模式？

1. 单例模式：DetectionService、KnowledgeRetriever、EmbeddingService、MultiAgentOrchestrator 等模块级单例；YOLO 模型缓存同一权重只加载一次
2. 分层架构：API → Agent/Service → Entity → Storage
3. 中间件模式（洋葱模型）：CORS → 请求日志 → 限流
4. 策略模式：LLM 后端可切换（Qwen/OpenAI 兼容）；Embedding 凭据三级优先；检索双策略（pgvector/词法）
5. 状态图模式：LangGraph StateGraph 做规划路由，AgentState 共享状态
6. 生产者-消费者模式：并行专家把事件写入共享 asyncio.Queue，编排器消费合并成单一 SSE 流
7. 降级模式（Fallback）：Redis→内存、向量→词法、LLM 规划→正则→关键词、ffmpeg→原始编码，全链路优雅降级

#### Q34：Docker Compose 编排了哪些服务？为什么使用 Docker？

**两套编排文件：**

- 根目录 `docker-compose.yml`（基础设施，3 服务）：PostgreSQL（`pgvector/pgvector:pg15` 镜像，自带向量扩展）、Redis 7、MinIO
- `docker-compose.full.yml`（全栈部署，5 服务）：在上述基础上增加 backend（FastAPI）和 web（前端 + Caddy）

**使用 Docker 的优势：**

1. 环境一致性：pgvector 这类带扩展的数据库用官方镜像一键起，避免手工编译扩展
2. 一键部署：`docker-compose up -d`
3. 隔离性与健康检查：服务独立容器，healthcheck 保证就绪顺序
4. Volume 持久化数据库与对象存储数据

开发机上不装 Docker 时，基础设施跑在远程主机，本地后端通过 `.env` 配置连接——配置与代码分离使两种形态无缝切换。

### 3.1 Python 与后端开发

#### Q35：Python 的 asyncio 是什么？在项目中是如何使用的？

asyncio 基于事件循环与协程实现单线程非阻塞并发：`async def` 定义协程，`await` 让出控制权。

**本项目的使用场景：**

1. SSE 流式对话：`async for` 迭代 `astream_events`，chunk 到达即推送
2. 多专家并行编排：`asyncio.create_task` 并发执行专家，`asyncio.Queue` 合并事件流，`asyncio.Event` 实现子任务依赖等待，非流式路径用拓扑分波 + `asyncio.gather`
3. WebSocket 摄像头检测的收发循环

**asyncio 与多线程/多进程的分工（这是本项目的重点实践）：**

- I/O 密集（LLM 调用、SSE 推送）→ asyncio
- 长耗时后台任务（训练、视频检测）→ `threading.Thread(daemon=True)`
- CPU 密集且要避免 GIL 影响事件循环（模型评估）→ `ProcessPoolExecutor(spawn)` 独立进程

#### Q36：FastAPI 的依赖注入是什么？在项目中怎么用的？

通过 `Depends()` 声明依赖，框架自动创建、注入并管理生命周期。

**本项目的典型用法：**

```python
# 数据库会话（yield 依赖，请求结束自动 close）
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

# 当前用户认证 → 权限校验依赖链
@router.post("/stream")
async def chat_stream(..., user: User = Depends(require_permission("chat:use")),
                      db: Session = Depends(get_db)):
    ...
```

**优势**：资源生命周期自动管理；依赖链自动解析（权限依赖 → 用户依赖 → OAuth2 token 解析）；测试时可 override 依赖（本项目 pytest 用 SQLite 覆盖 `get_db`）。

#### Q37：SQLAlchemy ORM 是什么？项目中是如何使用的？

ORM 把表映射为 Python 类，用对象操作代替手写 SQL。

**本项目：**

```python
class DetectionTask(Base):
    __tablename__ = "detection_tasks"
    id = Column(Integer, primary_key=True)
    user_id = Column(Integer, ForeignKey("users.id"))
    task_type = Column(String(20))   # single/batch/video/camera
    status = Column(String(20))
    ...
```

- 16 张表集中在 `app/entity/db_models.py`，配合 Alembic 管理迁移
- 测试用 SQLite 替换 PostgreSQL（`tests/conftest.py`），所以 pgvector、JSON 高级特性要在真实库上验证
- 向量表 `knowledge_embeddings` 是例外：pgvector 的 vector 类型用原生 SQL 建表和查询，说明 ORM 与原生 SQL 可以按需混用

#### Q38：Vue 3 的 Composition API 和 Options API 有什么区别？

#### Q39：Pinia 是什么？和 Vuex 有什么区别？

#### Q40：前端是如何实现 SSE 流式接收的？

用 fetch + ReadableStream（而非 EventSource）：

```javascript
const response = await fetch('/api/chat/stream', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
  body: JSON.stringify({ message, session_id, attachments })
})
const reader = response.body.getReader()
const decoder = new TextDecoder()
let buffer = ''
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  buffer += decoder.decode(value, { stream: true })
  // 按 \n\n 切分事件，逐条处理 data: 前缀行
  for (const data of parseSSE(buffer)) {
    switch (data.type) {
      case 'session':    saveSessionId(data.session_id); break
      case 'agent_route': showPlan(data); break
      case 'text_chunk': appendText(data.content); break
      case 'tool_call':  showToolCall(data); break
      case 'tool_result': renderToolResult(data); break  // 含标注图 base64
      case 'done':       finish(); break
    }
  }
}
```

**为什么不用 EventSource**：它只支持 GET、不能自定义 Authorization 头，而本项目需要 POST 传消息体和附件列表。注意点：TCP 分包可能把一条事件拆到两次 read，必须维护 buffer 按 `\n\n` 完整切分再 JSON.parse。

### 3.3 系统设计

#### Q41：如果要将这个项目部署到生产环境，你会怎么做？

项目已提供 `docker-compose.full.yml` 全栈编排（postgres/redis/minio/backend/web），生产化在此基础上：

1. 前端：`npm run build` 产物由 web 容器（Caddy）托管，自动 HTTPS
2. 后端：Uvicorn 多 worker；注意多进程下必须依赖真实 Redis（内存降级的记忆/限流不跨进程共享）
3. 反向代理：SSE 路径关闭缓冲（后端已带 `X-Accel-Buffering: no`）、WebSocket 升级配置、上传大小上限对齐（zip 100MB）
4. 数据层：PostgreSQL 连接池与定期备份；Redis 持久化；MinIO 纠删码
5. 密钥管理：`.env` 不入库，JWT secret / LLM API key 用环境变量或 secret 管理
6. GPU 节点部署推理（settings device 切 cuda，imgsz 恢复 640）
7. 监控告警 + 日志收集（现有 `logs/app.log` 轮转日志接入 Loki/ELK）
8. CI/CD：跑 pytest + Vitest → 构建镜像 → 部署

#### Q42：如何保证系统的高可用性？

1. **应用层**：多 worker/多实例 + `/api/health` 健康检查配合负载均衡摘除故障节点
2. **数据层**：PostgreSQL 主从、Redis Sentinel、MinIO 纠删码
3. **项目内已实现的降级能力（可重点讲）**：
   - Redis 不可用 → 记忆与限流自动降级进程内存，服务不中断
   - pgvector/Embedding 不可用 → 词法检索兜底，且向用户如实标注
   - LLM 规划失败 → 正则/关键词路由兜底
   - 多专家并行时单专家失败只影响自己的分节，不拖垮整条回复
   - ffmpeg 缺失 → 回退原始编码视频
4. **隔离**：CPU 密集评估放独立进程，训练/视频放后台线程，防止拖垮 API 事件循环
5. **限流**：按路径差异化限流保护重计算接口（视频 5 次/分钟）

#### Q43：如果检测请求量突然增大 10 倍，系统会怎样？你会如何优化？

**瓶颈分析：**

1. YOLO CPU 推理（数百 ms/张）是最大瓶颈
2. 数据库连接池耗尽
3. MinIO 上传带宽
4. LLM API 限流与延迟

**优化方案：**

短期应急：

- 限流中间件已按路径差异化限流（检测类接口 5~30 次/分钟），先顶住流量返回 429
- 增加 Uvicorn worker；把同步检测改为任务队列（Celery/RQ）异步化

中期：

- GPU 推理（吞吐提升 10-50 倍）；TensorRT/ONNX + FP16/INT8 量化；批推理
- 数据库读写分离；标注图直接走 MinIO 预签名 URL 减少后端 I/O

长期：

- 检测服务独立成推理微服务，K8s 水平扩缩容
- 模型服务化（Triton Inference Server）统一管理版本与批调度

### 3.4 项目深度追问

#### Q44：对话记忆是如何实现的？

三层记忆设计（`app/agent/memory.py` + `chat.py`）：

1. **Redis 短期记忆**：`ConversationMemory` 按 `chat:session:{user_id}:{session_id}` 存最近 20 条消息，TTL 24 小时；附件路径也按会话保存（最多 10 个检测轮次），所以用户说"再检测一次"能找到之前的图
2. **数据库持久层**：`chat_sessions`/`chat_messages` 全量落库；Redis 记忆为空（过期/重启）时自动从 DB 回灌最近 20 条——短期记忆丢了也能续聊
3. **注入方式**：历史消息经 `chat_history` MessagesPlaceholder 进入专家的 prompt

**并行模式的一致性处理**：多专家并行时每个专家 `record_memory=False`，编排器在全部完成后把"用户消息 + 合并后的 assistant 文本"统一写一次记忆，避免多专家重复写、乱序写。

**Redis 的优势**：内存级读写、TTL 自动清理、多实例共享（配合无状态后端水平扩展）；不可用时降级进程内存保证开发可用。

#### Q45：MinIO 在项目中承担什么角色？为什么不用本地文件系统？

**承担角色：**

- 检测原图与标注图、视频检测的标注视频（`detections/{task_id}/...`）
- 聊天上传的原始附件（`chat-originals/{user_id}/...`）
- 训练导出的模型权重

**不用本地文件系统的原因：**

1. S3 兼容 API，未来可无缝迁移云对象存储
2. 多实例部署共享存储
3. 预签名 URL：`chat_messages.attachments` 只存 object_name，读历史时实时重签短期 URL——既避免 URL 过期失效，又不把永久直链暴露出去（这是本项目聊天历史附件能长期回看的关键设计）
4. 纠删码可靠性、内置 Console 管理

#### Q46：项目中如何处理错误和异常？

1. **全局异常处理器**（`app/core/` 注册到 FastAPI）：兜底捕获，返回统一错误体，不暴露堆栈；错误细节进 `logs/app.log`
2. **认证/权限/业务异常**：401/403/404 等语义化状态码；WebSocket 用自定义 close code（4401 未认证 / 4403 无权限）
3. **Agent 层降级而非报错**：单专家失败 → 分节内 ⚠️ 提示；全部失败才发 SSE `error` 事件；工具内部异常返回结构化错误文本给 LLM，让 LLM 向用户解释而不是流中断
4. **外部依赖异常**：Redis/pgvector/Embedding/ffmpeg 均有降级路径（见 Q42），异常留痕日志
5. **后台任务**：视频/训练线程内 try-except，把失败状态写进 Redis 进度或任务表，前端轮询可见
6. **前端**：Axios 拦截器统一处理错误码，token 过期跳转登录

#### Q47：项目中如何保证前后端的数据一致性？

1. **Schema 契约**：后端 Pydantic 2 定义请求/响应模型自动验证；改字段名必须前后端同步改（项目规范明确要求）
2. **共享映射表**：多智能体分节标题的 `AGENT_LABELS`（后端）与 `AGENT_NAME_MAP`（前端 `utils/toolChain.js`）必须保持同步——这是典型的跨端一致性约束
3. **统一 Axios 客户端**：`utils/request.js` 统一 baseURL `/api`、拦截器、错误处理
4. **事务与状态机**：检测任务创建→结果写入→提交在一个事务；异步任务用 processing/completed/failed 状态轮询同步进度
5. **唯一默认模型**：`model_versions.is_global_default` 设置时先清除旧默认，保证全局只有一个启用模型
6. **附件一致性**：DB 只存 MinIO object_name（稳定标识），展示 URL 每次实时签发，避免存过期 URL 造成的"历史图片裂图"

5. 数据：引入 DVC 管理多个数据集版本（fdd/fdd2/fdd3/fdd5 目前靠目录约定）

#### Q50：你对 AI Agent 的未来发展有什么看法？

1. 多模态 Agent：文本 → 图像/视频/语音的统一理解与生成（本项目的"发图即检测"就是初步的多模态入口）
